76 Applications
77 ------------
78 
79 Two applications are provided with this library. They are RemoteContext
80 and CPNKernel.
81 
82 ### CPNKernel
83 
84 This is a loader application which takes a set of command line options and
85 configuration files and loads up a CPN network. This expects that all the nodes
86 can be looked up with the node loader. (See Node Library section below)
87 
88 ### RemoteContext
89 
90 This is a realization of a centralized version of the remote context
91 daemon over sockets. See the output to the `-h` option for details on
92 how to run.

106 
107 ### Random Instruction
108 
109 There are two versions of the random instruction case study. Both are in
110 the `examples/RandomInstruction` directory. The first version
111 instantiates the random instruction nodes in a single process and is
112 named `RandomInstruction`. Run with `-h` to get a list of options.
113 
114 The second version exercises the remote queue/context functionality.
115 This version is called `RemoteRandomInstruction`. The `-h` option gives
116 some details on how to run it. The remote random instruction takes a
117 config file that tells it how to create notes on what processes and what
118 processes exist. See the `-h` option and the example config
119 `configk2n100i100` in the same directory for examples of the
120 configuration file. To run this first run the RemoteContext then
121 run each of the processes. For the example config we might run:
122 
123  Run on zeus: RemoteContext zeus 12345
124  Run: RemoteRandomInstruction configk2n100i100 kone zeus 12345
125  Run: RemoteRandomInstruction configk2n100i100 ktwo zeus 12345
126 
127 This would run the RemoteContext daemon on zeus then create two
128 instances of the remote random instruction each named kone and ktwo.

145 About External Endpoints
146 ------------------------
147 
148 External endpoints are a special handle to a queue endpoints that is not
149 associated with a node. This is useful for the case where you want to
150 input or output data to the network of nodes. For example, instantiate
151 some network of nodes, then create two endpoints to be input and output.
152 Then this allows for the application to grab endpoints outside of a node
153 for putting data in and out. One has to be careful though as all the
154 same rules apply for determinism and deadlock. The idea behind this was
155 that CPN could be used in a larger application where CPN is just a
156 library used to compute some intermediate value for the application. One
157 of the design goals in mind while writing CPN was that it should be
158 usable in as many situations as possible.
159 
160 Example:
161 
162  kernel.CreateExternalWriter("writer name");
163  ... generate queues...
164  CPN::OQueue<int> out = kernel.GetExternalOQueue("writer name");
165  while (data) {
166  ...
167  out.Enqueue(...);
168  ...
169  }
170  out.Release();
171  ...
172 
173  kernel.DestroyExternalEndpoints("writer name");

175 About Function Nodes
176 --------------------
177 
178 These are nodes that can be created by passing a function or function
179 object to the kernel. It is setup with templates to allow for the
180 passing of up to 9 parameters to a function node. The idea here is that
181 sometimes one wants a simple node where writing all the machinery for a
182 full blown node would be overkill. This feature is used in the tests in
183 several places where writing a full node would have been painfully
184 verbose.
185 
186 Example:
187 
188  void testnode(CPN::NodeBase * node, std::string endpoint) {
189  CPN::IQueue<int> in = node->GetIQueue(endpoint);
190  ... read from in
191  }
192 
193  ...
194  in setup do
195  ...
196  kernel.CreateFunctionNode("node name", testnode, std::string("endpoint name"));

198 Node Library
199 ------------
200 
201 One of the features is dynamic node type loading. The way this works is
202 that there is a node loader class which has a list of places to search
203 for nodes. The first thing the node loader does is look through the
204 symbol table of the program for a symbol which is a mangling of the node
205 type name. If this symbol is found it will call it as a function to get
206 a node factory for that node type. If it is not found then it will look
207 into an internal dictionary to see if some dynamic library can be loaded
208 which will provide the symbol. The way the node loader knows if there is
209 a library it can load is by telling it with a node list file. The node
210 list file has a very simple syntax. There are two kinds of directives,
211 include and defining a node type name and its library. The include is
212 simply the keyword include followed by arbitrary white space then a list
213 of files to include separated by white space. All commands end in a
214 semicolon. The library node type name is the keyword lib followed by
215 white space then the name of the dynamic library to load followed by
216 white space then the mangled node type name and finished with a
217 semicolon. By default the node loader will look into \${prefix}/cpnnodes
218 for files matching `*.nodelist` and load them as additional node lists.
219 
220 There are some default nodes contained in the `nodelib` directory.

222 Defining A Node
223 ---------------
224 
225 To define a node, subclass `CPN::NodeBase`. Then somewhere in the
226 implementation file put the macro
227 `CPN_DECLARE_NODE_FACTORY(typename, classname);`. Note that typename
228 must be a valid as a symbol name. There is also a macro
229 `CPN_DECLARE_NODE_AND_FACTORY(typename, classname);` which allows one to
230 declare the node and factory all at once and all that is required is a
231 definition of the `classname::Process()` function. This allows stateless
232 nodes to be created more easily.
